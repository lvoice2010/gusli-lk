# Задание для Claude Code — клиентский личный кабинет CyberGusli: дашборд отчётности + правки ЛК

## 1. Контекст
Личный кабинет EngineGusli уже существует (раздел `ai_admin2`: Дашборд, Ассистенты, Звонки, Финансы, Документы, Глоссарий, Тарифы). Клиенты, которым мы настроили ассистента и которые оплатили, хотят видеть **статистику работы ассистента**. Текущий дашборд показывает «сколько звонков и сколько потрачено», но не отвечает на главные вопросы клиента: сколько ИИ закрыл сам, сколько ушло на оператора, о чём звонят и сколько это экономит.

Требования собраны из: UX-аудита текущего дашборда, анализа переписок с 4 клиентами (Русич — недвижимость/Bitrix24; Sage и Рыба моя — рестораны/бронирование; Медок — медклиника/статистика), и плана развития 2026 (интеграции Bitrix24, amoCRM, YClients, Restoplace).

## 2. Цель
Дать клиенту понятный **отчётный дашборд**, который отвечает на: сколько обращений, сколько ИИ закрыл сам vs передал оператору (и почему), о чём звонят, сколько это экономит, и что ассистент сделал в его CRM. Приоритет доступа — действующие оплатившие клиенты; затем доступно всем (и self-serve, и кому настраиваем мы).

## 3. Перед стартом (агенту обязательно)
- Изучить существующий репозиторий ЛК: стек, текущие виджеты дашборда (Конверсия, Динамика звонков, Расходы, Использование ассистентов), как тянутся данные по звонкам (модель звонка, поля, API).
- Сохранить текущий стек, стиль и компоненты графиков — переиспользовать, не вводить новый UI-фреймворк.
- Не ломать существующие разделы (Ассистенты, Звонки, Финансы).

## 4. Объём работ

### A. Новый дашборд отчётности (P0)
Все виджеты считаются за выбранный период.

1. **Единый переключатель периода** (Сегодня / Вчера / Неделя / Месяц / Произвольный) — управляет всем дашбордом. Сравнение с предыдущим **сопоставимым** периодом (для «сегодня» — к этому же времени вчера, не к полному дню).
2. **Сводка KPI**: всего обращений; закрыто ИИ; передано оператору (эскалации) и % эскалаций; средняя длительность (точность до секунд); стоимость за период.
3. **ИИ vs оператор (эскалации)**: доля закрытых ИИ против переданных оператору, с разбивкой по причинам перевода.
4. **Классификация обращений**: целевые / повторные / купившие / нецелевые (спам, агенты, предложения).
5. **Тематика обращений**: топ тем «о чём звонят» + динамика.
6. **Воронка**: обращения → дозвонились → целевые → лиды/брони/записи. У каждого шага — тултип с определением (что именно считается).
7. **ROI / экономия**: закрытые ИИ обращения → сэкономленные минуты/часы оператора, стоимость диалога, экономия за период (по ставке оператора из настроек).
8. **Статус интеграций**: сколько действий выполнено в CRM за период (карточек в Bitrix24 / броней в Restoplace / записей в YClients / сделок в amoCRM). Показывать только для **подключённых** интеграций.
9. **Кнопка «Скачать отчёт» (PDF)** — сводка за период под презентацию руководству клиента.

### B. Правки раздела «Звонки»
- KPI-сводка над таблицей (как в п. A.2).
- Колонки **«Тема»** и **«Эскалация»** (да/нет + причина).
- Расшифровка статуса «Успех/Отказ» (тултип) или замена на понятные статусы.
- Явная кнопка **«Открыть»** в строке; вся строка надёжно кликабельна (сейчас клик по ячейке выделяет текст вместо открытия карточки).

### C. Правки ЛК
- Исправить пункт меню **«Голосарий» → «Глоссарий»** (роут `/glossary` оставить).
- Подготовить **перенос ЛК на основной домен `cybergusli.ru`**: роутинг, абсолютные ссылки, ассеты, куки/CORS, редирект со старого адреса. Согласовать с DevOps.
- Фильтр **«тест / боевые ассистенты»**, чтобы тестовые сущности (например «Тестовый ассистент Муха», «русич тестовый 2») не попадали в клиентскую статистику.

## 5. Данные и зависимости (важно)
Эти поля нужны от бэкенда/моделей — без них виджеты делать с честным пустым состоянием, **не выдумывать цифры**:
- Флаг **эскалации** + причина перевода на каждом звонке.
- **Классификация** обращения (целевой / повтор / купивший / нецелевой).
- **Тематика** (LLM-тег темы).
- Связь звонка с результатом в CRM (id карточки / брони / записи).
- **Ставка оператора** клиента (для ROI) — в настройках.

Если данных пока нет — виджет с пустым состоянием и подписью вида «данные появятся после подключения …».

## 6. Чего НЕ делаем сейчас (non-goals)
- Сами интеграции Bitrix24 / amoCRM / YClients / Restoplace — это отдельный трек плана 2026; здесь только **отображение статуса**, если интеграция подключена.
- Модули исходящего обзвона, уведомлений и QA — вне этой задачи.

## 7. Критерии приёмки
- Клиент выбирает период — весь дашборд пересчитывается; сравнения корректны (без «на 0 мин больше» и сравнения неполного дня с полным).
- Видны: эскалации (ИИ vs оператор) с причинами, классификация, тематика, воронка с тултипами, ROI/экономия, статус интеграций (для подключённых).
- В «Звонках»: колонки Тема и Эскалация, рабочая кнопка «Открыть», KPI-сводка.
- «Глоссарий» исправлен; тестовые ассистенты не искажают статистику.
- Нет выдуманных данных; пустые состояния честные.
- Существующий функционал не сломан.

## 8. Приоритеты
- **P0**: переключатель периода, KPI-сводка, эскалации, классификация + тематика, ROI/экономия.
- **P1**: статус интеграций, PDF-отчёт, правки «Звонков», фильтр тест/боевые, перенос домена.
- **P2**: тултипы ко всем метрикам, почасовая карта нагрузки, ИИ-инсайты/рекомендации.
